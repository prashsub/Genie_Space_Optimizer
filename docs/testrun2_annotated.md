## How to Read This Document

This document traces a single end-to-end run of the **Lever Loop** — an automated optimization pipeline that iteratively improves a Databricks Genie Space's ability to answer natural-language questions with correct SQL. The logs don't explain the hidden logic connecting each phase, so this document fills in those gaps.

### The Big Picture (TL;DR)

```
┌─────────────────────────────────────────────────────────────────────┐
│  1. LOAD benchmarks (24 questions + ground-truth SQL)              │
│  2. PREPARE the Genie Space config (tables, columns, metadata)     │
│  3. RUN LEVER LOOP (iterative optimization):                       │
│     For each iteration:                                            │
│       a. Analyze failures → extract root causes (ASI)              │
│       b. Cluster failures by root cause                            │
│       c. Strategist proposes fixes (patches) targeting "levers"    │
│       d. Apply patches to Genie Space (column descriptions,        │
│          instructions, example SQLs)                               │
│       e. Wait for propagation (30s)                                │
│       f. Re-evaluate all 24 questions                              │
│       g. Accept or rollback based on accuracy delta                │
│     Repeat until accuracy target met or max iterations reached     │
└─────────────────────────────────────────────────────────────────────┘
```

### Key Concepts

| Term | Meaning |
|------|---------|
| **Genie Space** | A Databricks AI-powered SQL assistant configured with tables, columns, instructions, and examples to answer natural-language questions |
| **Lever** | A category of configuration change that can improve Genie's accuracy. Levers 1-5 correspond to: (1) Table & Column Descriptions, (2) Metric Views, (3) Functions, (4) Join Specifications, (5) Genie Space Instructions & Example SQLs |
| **Benchmark** | A set of question/ground-truth-SQL pairs used to measure accuracy |
| **ASI** | Automated Signal Intelligence — the system that extracts structured failure metadata (root cause, blame set, counterfactual fix) from each failed judge verdict |
| **Judge** | An automated evaluator that scores Genie's SQL on a specific dimension (syntax, schema, logic, completeness, etc.) |
| **Arbiter** | A meta-judge that makes the final call: `both_correct`, `genie_correct`, `ground_truth_correct`, or `neither_correct` |
| **Cluster** | A group of failures sharing the same root cause and blame set — used to address multiple failures with a single patch |
| **Action Group (AG)** | A bundle of proposed patches generated by the Adaptive Strategist to address one or more failure clusters |
| **Patch** | A concrete change applied to the Genie Space config (e.g., updating a column description, adding an instruction) |
| **Soft Signal** | A question that the arbiter marked as correct, but where one or more individual judges still flagged issues — indicates suboptimal quality even though the answer is technically right |

## Phase 5a: ASI Extraction (Root Cause Analysis)

> **What's happening:** For each failed question, the system extracts structured failure signals from every judge that flagged it. This is the "detective work" phase.

### Example: Hard Failure — Question `_gf_022`
*"How many holidays are there per year in our date dimension?"*

```
Judge: result_correctness | Verdict: FAIL
  ASI metadata: failure_type=wrong_aggregation
  Counterfactual: "Result mismatch: expected 0 rows, got 8 rows"
  Final root cause: wrong_aggregation

Dominant root cause: wrong_aggregation
Cluster group key: (wrong_aggregation, "(none)")
```

> **How to read this:** Genie used `SUM(CASE WHEN is_holiday = TRUE THEN 1 ELSE 0 END)` instead of `WHERE is_holiday = TRUE` with `GROUP BY year`. The result was a single aggregated row instead of a per-year breakdown. The ASI system identified this as a "wrong_aggregation" problem.

### Example: Hard Failure — Question `_005`
*"Show me monthly booking trends for the last 12 months"*

```
Judge: logical_accuracy | Verdict: FAIL
  failure_type: other
  blame_set: ['start_date', 'end_date', 'check_in_date', 'booking_created_date']
  counterfactual: "The TVF get_booking_trends should use a date range looking back
                   12 months from the current date"
  Final root cause: missing_filter

Judge: asset_routing | Verdict: FAIL
  failure_type: asset_routing_error
  blame_set: ['asset_routing:MV']
  counterfactual: "Add instruction to prefer MV for this query pattern"
  Final root cause: asset_routing_error

Dominant root cause: missing_filter
```

> **How to read this:** This question has **multiple overlapping failures**:
> 1. Genie routed to a TVF (Table-Valued Function) instead of the metric view
> 2. The TVF was called with a forward-looking date range instead of backward
> 3. Seven different judges flagged this question (it's the worst failure)
>
> The ASI system traces each judge's failure to a root cause. A "dominant root cause" is chosen by majority vote across all judges.
>
> **Key concept — blame_set:** This is a list of columns, tables, or configuration elements that are "blamed" for the failure. The strategist uses these to know exactly what to patch.

### Example: Hard Failure — Question `_001`
*"What is the total revenue by destination country this year?"*

```
Dominant root cause: missing_filter
Cluster group key: (missing_filter, "['check_in_date']|['this_year_filter']")
```

> **How to read this:** Genie interpreted "this year" as "last 12 months" (`ADD_MONTHS(CURRENT_DATE(), -12)`) instead of "since January 1st" (`DATE_TRUNC('year', CURRENT_DATE())`). This is a semantic interpretation error — the system needs better temporal interpretation instructions.

## Phase 6: The Lever Loop (Iterative Optimization)

> This is the main optimization loop. Each iteration follows this pattern:
>
> 1. **Adaptive Strategist** analyzes failures + past reflections → proposes an Action Group
> 2. **Proposals** are generated (column description updates, instructions, example SQLs)
> 3. **Validation** — proposals with invalid SQL are rejected
> 4. **Patches** are applied to the Genie Space via API
> 5. **Propagation wait** (30 seconds for changes to take effect)
> 6. **Evaluation** — all 24 questions re-evaluated
> 7. **Confirmation evaluation** — a second run to average out variance
> 8. **Accept or rollback** based on accuracy change

## Appendix: Judge Evaluation Format

Each question evaluation follows this template:

```
--- Q[N]: q[N] --------------------------------------------------------
| Question:  "[natural language question]"
|
| Genie SQL:
|   [SQL generated by Genie]
| Genie Status: MessageStatus.COMPLETED
| Genie Analysis: [Genie's natural language summary of results]
|
| Ground Truth SQL:
|   [Expected SQL from benchmark]
|
| Result Comparison: Match: YES/NO (method) | GT rows: N | Genie rows: N
|   GT hash: XXXX | Genie hash: XXXX
|   [Sample rows from both result sets]
|
| Judge Verdicts:
|   syntax_validity          PASS/FAIL  -- [rationale]
|   schema_accuracy          PASS/FAIL  -- [rationale]
|   logical_accuracy         PASS/FAIL  -- [rationale]
|   semantic_equivalence     PASS/FAIL  -- [rationale]
|   completeness             PASS/FAIL  -- [rationale]
|   response_quality         PASS/FAIL  -- [rationale]
|   asset_routing            PASS/FAIL  -- [rationale]
|   result_correctness       PASS/FAIL  -- [rationale]
|   arbiter                  [verdict]  -- [rationale]
```

### Judge Dimensions Explained

| Judge | What It Checks |
|-------|---------------|
| **syntax_validity** | Is the SQL syntactically correct? Can it parse and execute? |
| **schema_accuracy** | Does the SQL reference the correct tables and columns? |
| **logical_accuracy** | Does the SQL logic match what the question is asking? |
| **semantic_equivalence** | Are the Genie and ground-truth SQLs semantically equivalent (same meaning, even if different syntax)? |
| **completeness** | Does the SQL capture all aspects of the question (filters, groupings, etc.)? |
| **response_quality** | Is Genie's natural-language analysis of the results accurate and helpful? |
| **asset_routing** | Did Genie choose the right table/view/function to query? (e.g., metric view vs base table vs TVF) |
| **result_correctness** | Do the actual query results match between Genie and ground truth? (Row counts, hash comparison) |
| **arbiter** | Final verdict combining all judges: `both_correct`, `genie_correct`, `ground_truth_correct`, or `neither_correct` |

### Result Comparison Methods

| Method | Meaning |
|--------|---------|
| `identical_sql` | Genie produced exactly the same SQL as ground truth |
| `exact` | Different SQL, but results are identical (same rows, same hash) |
| `hash` | Results match by hash comparison (faster than row-by-row) |
| `signature` | Results match structurally (same shape, compatible values) |
| `mismatch` | Results differ — different row counts or different hashes |
